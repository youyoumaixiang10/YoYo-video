# 完整画面脚本合同

## 事实完整性门槛

输出具体分镜前，先确认用户提供的真实核心事件已经可以复述，至少能分清发生了什么、涉及谁、关键变化或结果是什么。信息不足时：

- 不得输出具体幕、补全故事或用“合理假设”代替事实。
- 不得补造事件、参与者、因果、动作或原话；即使标注“假设”也不允许。
- 每轮只问一个最高价值的必要问题，不得一次罗列多个问题。
- 缺少关键转折时，优先询问从起因到结果之间发生的关键变化。
- 用户记不清真实原话时，不得生成带引号的伪原话；可在事实足够后使用不带引号的旁白转述，或标记“原话待确认”。

只有核心事件达到可复述状态后，才进入下方输出顺序。

## 输出顺序

1. 判断故事是否适合拆成多条内容；需要选择方向时，提出不超过 3 个方向并推荐 1 个。
2. 明确本条发布标题建议、核心事件、核心情绪与最终表达。发布标题属于发布信息，除非用户明确同意，不得默认烧录进视频画面。
3. 按时间顺序输出 5—7 幕完整画面脚本；每一幕都使用下方原样字段，不得合并、改名或遗漏。
4. 输出全部幕之后，原样附上“整体验收门槛”。

## 无声叙事与因果衔接

- 画面脚本必须通过“字幕/对白”独立讲清核心故事。逐幕关闭旁白阅读时，观众仍应知道谁做了什么、为什么这样做、发生了什么变化。
- 旁白栏默认填写“无”。只有用户明确提出需要旁白时，才写旁白文本；即使启用旁白，也不得让它成为理解关键因果或转折的唯一渠道。
- 逐对检查相邻两幕是否形成自然的“起因—行动—回应—结果”推进。后一幕的提问、动作、情绪变化或场景切换，应能从前一幕自然推出。
- 若相邻两幕之间存在逻辑跳跃，只补充基于用户已提供事实的最短必要过渡；不得套用固定话术，也不得为了衔接而编造动机或因果。
- 真实原话构成笑点或故事回响时，必须逐句完整保留。多个连续问题应分别写出，不得为了省时合并成概括句。
- 每屏同时显示的叙事文字和对白不超过 3 行；长收束拆成顺序出现的多个阶段，不能把人物和字号一起缩小来容纳文字。

## 单幕固定格式

```markdown
## 第N幕｜幕名
时间：00:00—00:04
画面与构图：
人物动作与表情：
字幕/对白：
旁白：
动效与绘制阶段：
风险检查：
```

“风险检查”至少核对当前幕的故事事实、角色一致性、动作与情绪、无旁白时是否看得懂、因果是否突然、原话是否完整、文字是否超过 3 行，以及是否出现角色、画风、背景色或内容漂移。

## 整体验收门槛

以上是当前完整画面脚本。请逐幕确认故事、人物、动作、情绪、文字和结尾。只有你明确回复“确认画面脚本，可以生成”后，我才会开始正式出图和视频制作；对角色图、画风图或单幕预览的认可不等于整体确认。旁白默认不生成，只有你另行明确提出并确认旁白文本后才会配音。
